Add passwordless email login

Replace the password form with magic-link auth, reusing the existing session layer.

  • Draft
  • Scope: auth
  • Effort: ~2 days
  • 1 open question

Summary

Issue a signed, single-use token on email submit, email a link, and exchange the token for a session on click. No password storage, no new tables — tokens are stateless (JWT) and the existing session cookie is unchanged.

Problem

Password reset accounts for ~30% of support tickets and the bcrypt path is the slowest route in the app. Users already verify email at signup, so the password adds friction without adding much security.

Approach

Keep the current /login route and session middleware. Swap the password check for a two-step token flow: request → email link → verify. Tokens are HMAC-signed and carry the email + expiry, so no server-side store is needed.

Risks & trade-offs

Risks

  • Email deliverability — a delayed link blocks login entirely. Mitigate with a visible resend + clear "check spam" copy.
  • Stateless tokens can't be revoked before expiry. Accepted for a 15-min window; revisit if abuse appears.
